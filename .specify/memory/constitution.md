# Holigay Vendor Market Constitution

## Core Principles

### I. Code Quality & Type Safety (NON-NEGOTIABLE)

The server-action pattern, RBAC enforcement, and RLS posture proven across
`src/lib/actions/`, `src/middleware.ts`, and `supabase/migrations/` are the trust model of this
application. Regressions here undermine the whole platform, so these rules are hard gates, not
guidelines.

- TypeScript `strict: true` MUST remain enabled (`tsconfig.json`). Production code MUST NOT use
  `any`, `@ts-ignore`, or `@ts-expect-error`. Narrow via generics, `unknown` + type guards, or
  Zod inference instead.
- `src/types/database.ts` is generated by `npm run db:types` / `db:types:dev` and MUST NOT be
  hand-edited. Supabase clients MUST be instantiated as `createClient<Database>()`.
- Every server action in `src/lib/actions/` MUST:
  1. Validate input with a Zod schema from `src/lib/validations/` via `safeParse` before any
     database call.
  2. Authorize via `requireRole()` (`src/lib/auth/roles.ts`) at the
     top of the action body for any mutation or privileged read.
  3. Return a `{ success: boolean, error: string | null, data: T | null }` response — no
     throwing across the server/client boundary.
- Row Level Security MUST be enabled on every user-data table. New tables that store user or
  tenant data MUST ship with an RLS policy in the same migration that creates them.
- Database schema changes MUST be introduced as new, append-only, numbered migrations under
  `supabase/migrations/`. Previously-applied migrations MUST NOT be edited.
- CI (`.github/workflows/ci.yml`) MUST run `npm run format:check`, `npm run lint`, `npm test`,
  and `npm run build`, in that order. A red check on any of these blocks merge.
- Commented-out code, dead exports, and half-finished abstractions MUST be removed before
  merge. Known incomplete features (e.g., `src/lib/actions/team.ts`) MUST fail closed with an
  explicit error response and MUST be tracked as a spec under `specs/`.

### II. Testing Standards

Testing infrastructure exists (Vitest + jsdom + React Testing Library + `@testing-library/jest-dom`
via `vitest.config.ts` and `src/test/setup.ts`), but coverage is currently narrow. This
principle sets an honest current bar and a named path forward.

- `npm test` MUST pass on every PR; CI enforces this.
- Every new or modified server action in `src/lib/actions/` MUST ship with at least one
  happy-path test and one authorization-or-validation failure test, following the AAA
  (Arrange-Act-Assert) pattern and mocking the Supabase client as in
  `src/test/auth-roles.test.ts`.
- Every new or modified user-facing form in `src/components/forms/` MUST ship with a React
  Testing Library component test covering rendering, validation error display, and successful
  submission, following the pattern in `src/test/application-form.test.tsx`.
- Tests MUST use `@testing-library/user-event` for interaction; they MUST NOT assert against
  implementation details (class names, internal state, or DOM structure not visible to users).
- Reusable fixtures SHOULD be introduced under `src/test/fixtures/` once total test count
  exceeds ~20 files; until then, inlined mock data is acceptable.
- Integration coverage of the signup → `handle_new_user` trigger → profile-creation flow and
  the end-to-end application-submission flow (form → server action → email) SHOULD be added as
  this project matures; absence is a known gap, not a permitted regression.
- A Vitest coverage threshold is an aspirational gate. It MUST be introduced in
  `vitest.config.ts` once baseline server-action coverage reaches approximately 60%, at which
  point it becomes a MUST (tracked as `TODO(coverage-threshold)`).
- Email templates (`src/lib/email/templates/`) and RLS policies are presently untested.
  Template changes SHOULD be snapshot-tested; RLS changes SHOULD include a manual verification
  note in the PR until a DB-integration harness exists.

### III. User Experience Consistency

The Epic 6 brand re-skin (`#1C171C` background, Quicksand, `#A78BFA` primary, rainbow accents)
is load-bearing. Inconsistency anywhere silently undoes work done everywhere else.

- Colors and typography MUST be applied via the CSS tokens defined in `src/app/globals.css`
  (Tailwind v4 `@theme inline`). Inline hex/rgb values in JSX `style` props, `className`
  strings, or component files are prohibited. The existing violation in
  `src/app/layout.tsx:32-34` (Toaster `toastOptions`) is acknowledged and tracked as
  `TODO(layout-toaster-tokens)`; no new violations may be introduced.
- New UI MUST reuse primitives from `src/components/ui/` (button, input, textarea, select,
  badge, checkbox, card, spinner, file-upload) before a new component is introduced.
  Justification for a new primitive MUST appear in the PR description.
- Every form MUST use `react-hook-form` + `zod` + `@hookform/resolvers` and MUST wire
  `aria-invalid`, `aria-describedby`, and `role="alert"` on error messages. Labels MUST use
  `htmlFor` bound to an id generated via `useId()`.
- Every mutating server action's client caller MUST surface the result to the user via Sonner
  toast on success/failure, or via a route-level `error.tsx` when a full-page recovery is
  preferred. "Silent" successes are not permitted.
- Every route segment that performs data fetching MUST provide a `loading.tsx`. An `error.tsx`
  MUST be provided at any segment where the root `src/app/error.tsx` would deliver a
  worse-than-default UX.
- Minimum interactive-target size is 44×44 px; current `Button` primitive enforces this and
  MUST not be weakened.
- Accessibility tooling SHOULD be strengthened by adding `eslint-plugin-jsx-a11y` to
  `eslint.config.mjs`; until then, the aria/`role` patterns above remain MUST-enforced by
  reviewers.

### IV. Performance Requirements

The App Router is used correctly today (RSC-first, lean middleware matcher, minimal client
bundle). These rules keep it that way and close the cache-invalidation gap.

- Components default to React Server Components. A file MUST NOT be marked `'use client'`
  unless it uses React state, refs, effects, browser-only APIs, or third-party client-only
  libraries. RSC → Client conversions MUST be justified in the PR description.
- `src/middleware.ts` matcher MUST continue to exclude `_next/static`, `_next/image`,
  `favicon.ico`, and common image extensions. Broadening the matcher requires documented
  justification.
- Any mutating server action that changes data rendered by an existing route MUST call
  `revalidatePath()` (or set a `revalidate` export / use `unstable_cache` with appropriate
  tags) so the route re-fetches. Mutations that leave stale reads on screen are a defect.
- Any user-uploaded image displayed in the UI MUST be rendered via `next/image`. Raw `<img>`
  tags are permitted only for small brand SVGs or inline icons whose dimensions are known at
  build time.
- New runtime dependencies added to `src/app/layout.tsx` (or any module it imports) MUST be
  justified in the PR description with a stated reason and an expected bundle-size impact.
- Read-heavy dashboard views SHOULD use `unstable_cache` with invalidation tags to reduce
  Supabase round-trips; this is a SHOULD today and becomes a MUST once measured LCP on
  `/dashboard` exceeds 2.5 s.
- Performance budget: Largest Contentful Paint SHOULD be under 2.5 s for `/`, `/login`, and
  `/dashboard` on a modern laptop over a simulated 4G connection. Regressions that breach
  this budget MUST be either fixed or documented with a time-boxed exception.

## Additional Constraints

- **Stack lock-in** (changes require a constitutional amendment):
  - Framework: Next.js 16 App Router + React 19 + TypeScript.
  - Database & auth: Supabase (PostgreSQL with RLS), cookie-based sessions via `@supabase/ssr`.
  - Styling: Tailwind CSS v4 with `@theme inline` tokens in `src/app/globals.css`.
  - Forms & validation: `react-hook-form` + `zod` + `@hookform/resolvers`.
  - Email: Resend for transactional delivery.
  - Tests: Vitest + jsdom + React Testing Library + `@testing-library/jest-dom`.
  - Notifications: Sonner.
- **Environment contract**: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
  `RESEND_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, and `EMAIL_FROM_ADDRESS` are the documented
  variables in `CLAUDE.md`. New required env vars MUST be documented in `CLAUDE.md` in the
  same PR that introduces them. Secrets MUST NOT be committed.
- **Path alias**: `@/` resolves to `src/`. Relative imports that cross module boundaries
  (e.g., `../../lib/...`) MUST be rewritten to use the alias.
- **Migrations**: `supabase/migrations/` is append-only; numbered prefix is authoritative;
  applied in order. Local schema edits made directly against Supabase MUST be captured as a
  migration before the PR merges.
- **Logging**: `console.error` with contextual prefix is the current convention in server
  actions. Introducing a structured logging service (Sentry, Logflare, etc.) is an
  amendment, not a drop-in.

## Development Workflow & Quality Gates

1. **Task source of truth**: `specs/<nnn>-<slug>/tasks.md` tracks every task. PRs MUST
   reference the spec ID or task ID in the commit message (e.g.,
   `[001-consolidate-role-helpers]`, or `[6.8.2]` for historical Epic tasks archived in
   `docs/archive/TASKS.md`).
2. **Commit message convention**: `type(scope): description [task-id]`, matching existing
   history (e.g., `feat(ui): add toast notifications to all key server actions [6.8.2]`).
3. **Before opening a PR**, the author MUST locally run:
   - `npm run lint`
   - `npm run build`
   - `npm test`
   and MUST NOT rely on CI to discover failures that local runs would have caught.
4. **PR checklist** (reviewer-enforced, in addition to CI):
   - Principle I: every new mutating server action has a `requireRole()` guard
     and a `{ success, error, data }` return.
   - Principle II: new server actions and new user-facing forms are accompanied by tests.
   - Principle III: no inline hex colors; forms wire aria attributes; toasts or error
     boundary surface every mutation result.
   - Principle IV: new routes have `loading.tsx`; mutations call `revalidatePath()` where
     they affect existing views; no needless `'use client'`.
5. **CI gate order** (from `.github/workflows/ci.yml`) is authoritative: format → lint →
   test → build. A PR MUST NOT be merged with any gate red.
6. **Task completion**: a task in a spec's `tasks.md` MAY be marked complete only after
   (a) its acceptance criteria are met, (b) the gates above pass, and (c) the PR is merged.

## Governance

- This constitution supersedes ad-hoc practice. Where CLAUDE.md, README.md, or
  `.specify/templates/*.md` conflict with this file, this file wins and the other document
  MUST be amended to match in the next applicable PR.
- **Amendments** MUST be made via a PR that:
  1. Edits `.specify/memory/constitution.md` directly.
  2. Prepends (or updates) the Sync Impact Report HTML comment at the top of this file.
  3. Bumps the version footer according to semantic versioning:
     - **MAJOR**: backward-incompatible removal or redefinition of a principle or governance
       rule.
     - **MINOR**: a new principle/section is added, or existing guidance is materially
       expanded.
     - **PATCH**: clarifications, wording fixes, typo/formatting changes, non-semantic
       refinements.
  4. Updates `Last Amended` to the PR's merge date (ISO YYYY-MM-DD). `Ratified` never
     changes.
  5. Propagates any downstream edits required in `.specify/templates/*.md`, `CLAUDE.md`, or
     `README.md`, or records the deferral as a follow-up TODO in the Sync Impact Report.
- **Complexity justification**: introducing a new runtime dependency, a new abstraction layer,
  converting RSC to Client, or disabling an ESLint rule MUST be justified inline in the PR
  description. Reviewers MAY reject PRs whose complexity is not justified.
- **Exceptions** to MUST-rules in Principles I–IV MUST be time-boxed (≤ one release cycle)
  and tracked in a spec under `specs/`. Long-lived exceptions become amendments.
- **Runtime guidance**: `CLAUDE.md` at the repository root remains the day-to-day operational
  reference for contributors and assistants; it does NOT override this document.
